# Спецификация микросервиса «Отзывы и рейтинги» (Reviews & Ratings)

Документ описывает назначение, границы, сущности и API сервиса отзывов и рейтингов для проекта Fleur de Rêve. На основе этой спецификации строится OpenAPI-описание и дальнейшая реализация (например, на FastAPI).

---

## 1. Назначение и границы

### 1.1. Зачем нужен сервис

- **Единое место** для всех отзывов: о магазине в целом и о конкретных товарах.
- **Модерация**: отзывы публикуются только после одобрения (`is_approved`).
- **Рейтинги**: средняя оценка по магазину и по каждому товару для отображения в каталоге и на главной.
- **Независимость**: основной сайт (Django) и другие сервисы работают с отзывами только через API, без доступа к БД отзывов.

### 1.2. Что входит в контекст сервиса

| В зоне ответственности | Описание |
|------------------------|----------|
| Хранение отзывов       | Текст, автор (имя), оценка 1–5, привязка к товару или к магазину, дата создания. |
| Модерация              | Одобрение или отклонение отзыва (флаг `is_approved`). |
| Агрегация рейтингов    | Средний рейтинг по магазину; средний рейтинг по товару (по `product_id`). |
| Список отзывов         | Выдача с фильтрами (одобренные/все, по товару, по рейтингу, по дате) и пагинацией. |

### 1.3. Что остаётся за пределами сервиса

- **Пользователи и авторизация** — сервис не проверяет логин/пароль. Идентификатор пользователя (`user_id`), если передан, хранится только как метаданные (кто оставил отзыв). Аутентификацию выполняет вызывающая сторона (Django или API Gateway).
- **Каталог товаров** — сервис не проверяет существование `product_id`. Доверяем вызывающей стороне или договариваемся о валидации на уровне API (404 при неизвестном товаре — по желанию).
- **Уведомления** (например, «ваш отзыв одобрен») — отдельный сервис уведомлений, не часть Reviews.

## 2. Сущности и модели данных

### 2.1. Отзыв (Review)

Один отзыв от одного автора в один момент времени.

| Поле           | Тип      | Обязательное | Описание |
|----------------|----------|--------------|----------|
| `id`           | integer  | да (система) | Уникальный идентификатор отзыва. |
| `product_id`  | integer  | нет          | ID товара в каталоге. Если `null` — отзыв о магазине в целом. |
| `user_id`     | integer  | нет          | ID пользователя (из сервиса авторизации), если отзыв оставил авторизованный пользователь. |
| `name`        | string   | да           | Имя автора (как подпись к отзыву), макс. 100 символов. |
| `text`        | string   | да           | Текст отзыва. |
| `rating`      | integer  | да           | Оценка от 1 до 5 звёзд. |
| `is_approved` | boolean  | да           | Одобрен ли отзыв модератором. По умолчанию `false`. |
| `created_at`  | datetime | да (система) | Дата и время создания (ISO 8601). |

Ограничения:

- `rating` только 1, 2, 3, 4 или 5.
- `name` и `text` — не пустые после обрезки пробелов.

### 2.2. Агрегат рейтинга (Rating summary)

Не хранимая сущность, а результат расчёта по отзывам.

| Поле            | Тип     | Описание |
|-----------------|---------|----------|
| `average`       | number  | Среднее арифметическое оценок (1–5), округлённое до 1–2 знаков. |
| `count`         | integer | Количество учтённых отзывов (только одобренные или все — по договорённости). |
| `product_id`    | integer \| null | Если `null` — рейтинг по магазину; иначе — по товару с данным ID. |

Для отображения на сайте обычно считаем только одобренные отзывы (`is_approved = true`).

## 3. Сценарии использования

### 3.1. Пользователь оставляет отзыв

1. На сайте (Django) пользователь заполняет форму: имя, текст, оценка 1–5, опционально выбран товар.
2. Django (или фронт через BFF) отправляет в сервис `POST /reviews` с телом: `name`, `text`, `rating`, опционально `product_id`, опционально `user_id`.
3. Сервис проверяет данные, сохраняет отзыв с `is_approved = false`, возвращает 201 и созданный объект отзыва (включая `id`, `created_at`).

### 3.2. Просмотр списка отзывов на сайте

1. Запрос к сервису: например, `GET /reviews?approved_only=true&limit=10&offset=0`.
2. Опционально: фильтр по товару `product_id`, по рейтигу `min_rating`, сортировка по дате.
3. Сервис возвращает массив отзывов и при необходимости `total` для пагинации (по API_STANDARDS: limit/offset).

### 3.3. Модерация отзыва (админ/оператор)

1. Модератор запрашивает список неодобренных: `GET /reviews?approved_only=false` (или отдельный эндпоинт для очереди модерации).
2. Одобрение: `PATCH /reviews/{reviewId}` с телом `{ "is_approved": true }`.
3. Отклонение: `PATCH /reviews/{reviewId}` с телом `{ "is_approved": false }` (или удаление — см. ниже).

### 3.4. Отображение рейтинга на главной и в каталоге

1. **Рейтинг по магазину** (глобальный): `GET /ratings/summary` или `GET /reviews/rating-summary` без параметра товара (или с параметром `scope=shop`).
2. **Рейтинг по товару**: `GET /ratings/summary?product_id=42` (или путь вида `/products/42/rating-summary`).
3. Ответ: `{ "average": 4.5, "count": 120 }` (и при необходимости `product_id`). На сайте показываем звёзды и число отзывов.

## 4. Эндпоинты API (сводка)

| Метод   | Путь                          | Назначение |
|---------|--------------------------------|------------|
| GET     | `/reviews`                    | Список отзывов с фильтрами и пагинацией. |
| POST    | `/reviews`                    | Создать отзыв. |
| GET     | `/reviews/{reviewId}`         | Получить один отзыв по ID. |
| PATCH   | `/reviews/{reviewId}`         | Частичное обновление (модерация: `is_approved`). |
| DELETE  | `/reviews/{reviewId}`         | Удалить отзыв (по желанию — только для модерации/удаления спама). |
| GET     | `/ratings/summary`            | Средний рейтинг и количество отзывов (по магазину или по `product_id`). |

Детали параметров, тел запросов и ответов — в OpenAPI (YAML).

## 5. Параметры запросов и форматы

### 5.1. Список отзывов (GET /reviews)

- **Пагинация** (по API_STANDARDS): `limit` (по умолчанию 20, макс. 100), `offset` (по умолчанию 0).
- **Фильтры:**
  - `approved_only` (boolean): если `true` — только одобренные; если `false` — только неодобренные; если не передан — все.
  - `product_id` (integer): только отзывы по данному товару; если не передан — все (и по магазину, и по товарам). Для «только отзывы о магазине» можно ввести значение вроде `product_id=0` или отдельный параметр `scope=shop`.
  - `min_rating` (integer 1–5): только отзывы с оценкой не ниже указанной.
  - `max_rating` (integer 1–5): только отзывы с оценкой не выше указанной (опционально).
- **Сортировка:** `sort` = `created_at_desc` (по умолчанию) или `created_at_asc`, при необходимости `rating_desc` / `rating_asc`.

В ответе — массив отзывов и при необходимости поле `total` (общее количество записей по заданным фильтрам) для постраничной навигации.

### 5.2. Создание отзыва (POST /reviews)

Тело запроса (JSON):

- `name` (string, обязательно)
- `text` (string, обязательно)
- `rating` (integer 1–5, обязательно)
- `product_id` (integer, опционально) — если не передан, отзыв считается отзывом о магазине
- `user_id` (integer, опционально) — ID пользователя в основной системе

Ответ 201: созданный объект отзыва (включая `id`, `created_at`, `is_approved = false`). Ошибки 400/422 при невалидных данных.

### 5.3. Модерация (PATCH /reviews/{reviewId})

Тело запроса (JSON): объект с полями для обновления, минимум одно:

- `is_approved` (boolean) — одобрить или отклонить отзыв.

Ответ 200: обновлённый объект отзыва. 404 — отзыв не найден.

### 5.4. Рейтинг (GET /ratings/summary)

- **Параметр** `product_id` (integer, опционально): если передан — рейтинг по этому товару; если не передан — рейтинг по магазину (все отзывы без `product_id` или с `product_id = null`).
- **Параметр** `approved_only` (boolean, по умолчанию `true`): учитывать только одобренные отзывы при расчёте среднего и счётчика.

Ответ 200: `{ "average": 4.5, "count": 120, "product_id": 42 }` или для магазина `{ "average": 4.7, "count": 350, "product_id": null }`. Если отзывов нет — `count: 0`, `average` можно вернуть `null` или не включать.

## 6. Коды ответов и ошибки

- **200** — успешный GET или PATCH.
- **201** — успешное создание отзыва (POST).
- **204** — успешное удаление (DELETE), тело пустое.
- **400** — некорректный запрос (например, неверный формат или невалидное значение `rating`).
- **404** — отзыв или ресурс не найден (например, GET/PATCH/DELETE по несуществующему `reviewId`).
- **422** — ошибка валидации (например, пустой `text` или `name`).

Формат ошибки — по API_STANDARDS: объект с полями `error` (строка) и `code` (число, HTTP-код). При необходимости добавляется `details` (массив или объект с полями ошибок валидации).

## 7. Связь с другими частями системы

- **Django (основной сайт):**
  - На странице отзывов: GET `/reviews?approved_only=true&limit=...&offset=...`.
  - На странице товара: GET `/reviews?product_id=...&approved_only=true`, GET `/ratings/summary?product_id=...`.
  - Отправка формы отзыва: POST `/reviews` с `name`, `text`, `rating`, опционально `product_id`, опционально `user_id` (из текущего пользователя).
- **Админка/модерация:** GET `/reviews?approved_only=false`, PATCH `/reviews/{id}` для одобрения/отклонения, при необходимости DELETE.
- **Каталог/главная:** только GET `/ratings/summary` и GET `/ratings/summary?product_id=X` для отображения звёзд и количества отзывов.

После утверждения этой спецификации следующий шаг — описание API в OpenAPI (YAML) и реализация сервиса (например, на FastAPI).
