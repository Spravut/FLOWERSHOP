# Полное описание API сервиса «Отзывы и рейтинги»

Документ описывает каждый путь, схему и тип ответа из файла **openapi/reviews-service.yaml** — зачем они нужны и что делают.

---

## Общее назначение файла

**reviews-service.yaml** — это OpenAPI 3.0 спецификация микросервиса отзывов и рейтингов для магазина Fleur de Rêve. В одном месте задано:

- какие HTTP-методы и URL доступны;
- какие параметры и тела запросов принимаются;
- какие данные возвращаются и в каком формате;
- какие ошибки возможны.

По этому файлу можно генерировать документацию (Swagger UI), писать клиентский код и реализовывать сам сервис (например, на FastAPI).

**Базовый URL сервера в спецификации:** `http://localhost:8000` (для разработки).

---

## Пути (paths) и эндпоинты

Ниже разобраны все пути из раздела `paths` по одному.

---

### 1. `GET /reviews` — список отзывов

**Зачем нужен:**  
Чтобы показать пользователям отзывы на странице «Отзывы» или отзывы по конкретному товару на странице товара. Также используется модератором, чтобы просмотреть все отзывы или только очередь на модерацию.

**Что делает:**  
Возвращает список отзывов с учётом фильтров и пагинации. Ответ — объект с полями `items` (массив отзывов) и `total` (общее количество записей по выбранным фильтрам), чтобы можно было построить постраничную навигацию.

**Параметры запроса (все опциональные):**

| Параметр       | Тип    | По умолчанию   | Зачем нужен |
|----------------|--------|----------------|-------------|
| `limit`        | integer| 20             | Сколько отзывов вернуть на одной «странице» (макс. 100). |
| `offset`       | integer| 0              | Смещение: с какого по счёту отзыва начать (для 2-й, 3-й страницы и т.д.). |
| `approved_only`| boolean| —              | `true` — только одобренные (для публичной страницы), `false` — только неодобренные (очередь модерации). Не передавать — все отзывы. |
| `product_id`   | integer| —              | Только отзывы по товару с этим ID. Не передавать — все отзывы (и о магазине, и по товарам). |
| `min_rating`   | integer| —              | Только отзывы с оценкой не ниже указанной (1–5). |
| `max_rating`   | integer| —              | Только отзывы с оценкой не выше указанной (1–5). |
| `sort`         | string | `created_at_desc` | Как сортировать: по дате создания (сначала новые или старые) или по оценке (сначала высокие или низкие). |

**Ответ при успехе:** код **200**, тело — схема **ReviewList** (объект с полями `items` и `total`).

---

### 2. `POST /reviews` — создать отзыв

**Зачем нужен:**  
Пользователь на сайте заполнил форму отзыва (имя, текст, оценка, опционально выбор товара) — основной сайт (Django) отправляет эти данные в микросервис, чтобы отзыв сохранился в одном месте и потом проходил модерацию.

**Что делает:**  
Принимает JSON с полями отзыва (без `id`, без `is_approved`, без `created_at` — они выставляются сервисом). Создаёт запись с `is_approved = false`. Возвращает уже созданный полный объект отзыва (включая `id` и `created_at`), чтобы фронт мог показать «Спасибо, ваш отзыв отправлен на модерацию».

**Тело запроса:** схема **ReviewCreate** (обязательные поля: `name`, `text`, `rating`; опционально: `product_id`, `user_id`).

**Ответы:**

- **201** — отзыв создан, в теле — полный **Review**.
- **400** — некорректный запрос (BadRequest).
- **422** — ошибка валидации (например, пустой текст, оценка не от 1 до 5) — ValidationError, в теле может быть поле `details` с ошибками по полям.

---

### 3. `GET /reviews/{reviewId}` — получить один отзыв по ID

**Зачем нужен:**  
Когда нужно показать или отредактировать один конкретный отзыв (например, в админке модерации при открытии карточки отзыва).

**Что делает:**  
Возвращает один объект отзыва по его числовому идентификатору `reviewId` в пути.

**Параметры пути:**

- `reviewId` (обязательный) — целое число, ID отзыва.

**Ответы:**

- **200** — отзыв найден, в теле — **Review**.
- **404** — отзыва с таким ID нет (NotFound).

---

### 4. `PATCH /reviews/{reviewId}` — обновить отзыв (модерация)

**Зачем нужен:**  
Модератор одобряет или отклоняет отзыв. Достаточно обновить одно поле — `is_approved` (true/false), не трогая остальные данные. Для этого и выбран PATCH (частичное обновление), а не PUT.

**Что делает:**  
Принимает в теле JSON с полями, которые нужно изменить (минимум одно). В текущей схеме там только `is_approved`. Сервис обновляет только переданные поля и возвращает обновлённый отзыв.

**Параметры пути:**

- `reviewId` (обязательный) — ID отзыва.

**Тело запроса:** схема **ReviewUpdate** (сейчас одно поле: `is_approved` типа boolean).

**Ответы:**

- **200** — отзыв обновлён, в теле — полный **Review**.
- **404** — отзыв не найден.
- **422** — ошибка валидации переданных данных.

---

### 5. `DELETE /reviews/{reviewId}` — удалить отзыв

**Зачем нужен:**  
Удаление отзыва, например при модерации спама или по запросу пользователя/юриста. Полное удаление из хранилища.

**Что делает:**  
Удаляет отзыв с указанным ID. При успехе тело ответа пустое.

**Параметры пути:**

- `reviewId` (обязательный) — ID отзыва.

**Ответы:**

- **204** — отзыв удалён (тела нет).
- **404** — отзыва с таким ID нет.

---

### 6. `GET /ratings/summary` — средний рейтинг и количество отзывов

**Зачем нужен:**  
Показать на сайте «среднюю оценку» и «сколько отзывов» — на главной (рейтинг по магазину) или на карточке товара (рейтинг по этому товару). Один лёгкий запрос вместо загрузки всех отзывов и расчёта на клиенте.

**Что делает:**  
Считает по сохранённым отзывам среднее арифметическое оценок (1–5) и количество отзывов. Можно запросить рейтинг по всему магазину (без привязки к товару) или по одному товару. По умолчанию в расчёт попадают только одобренные отзывы.

**Параметры запроса (оба опциональные):**

| Параметр       | Тип    | По умолчанию | Зачем нужен |
|----------------|--------|--------------|-------------|
| `product_id`   | integer| —            | Если передан — считаем рейтинг только по отзывам этого товара. Не передавать — рейтинг по магазину (все отзывы без привязки к товару или с `product_id = null`). |
| `approved_only`| boolean| true         | Учитывать только одобренные отзывы при расчёте среднего и счётчика. |

**Ответ при успехе:** код **200**, тело — схема **RatingSummary** (`average`, `count`, при необходимости `product_id`). Если отзывов нет, `count = 0`, `average` может быть `null`.

## Модели данных (components/schemas)

Кратко, зачем нужна каждая схема и что в ней лежит.

---

### Review

**Назначение:** полная модель одного отзыва — то, что хранится в сервисе и возвращается в ответах GET (один отзыв или элементы списка).

**Поля:**

- `id` — уникальный идентификатор (целое число).
- `product_id` — ID товара, к которому привязан отзыв; `null` — отзыв о магазине в целом.
- `user_id` — ID пользователя в основной системе (если отзыв от авторизованного); может быть `null`.
- `name` — имя автора (как подпись), до 100 символов.
- `text` — текст отзыва.
- `rating` — оценка от 1 до 5 (звёзды).
- `is_approved` — одобрен ли отзыв модератором (по умолчанию false до модерации).
- `created_at` — дата и время создания в формате ISO 8601.

Все перечисленные поля в схеме помечены как обязательные в ответе.

---

### ReviewCreate

**Назначение:** тело запроса для создания отзыва (POST /reviews). Без служебных полей `id`, `is_approved`, `created_at` — их выставляет сервис.

**Поля:**

- `name` (обязательное) — имя автора.
- `text` (обязательное) — текст отзыва.
- `rating` (обязательное) — оценка 1–5.
- `product_id` (опционально) — если указан, отзыв привязывается к товару; иначе это отзыв о магазине.
- `user_id` (опционально) — ID пользователя в основной системе.

---

### ReviewUpdate

**Назначение:** тело запроса для частичного обновления отзыва (PATCH /reviews/{reviewId}). Сейчас используется в основном для модерации: передаётся только то, что нужно изменить.

**Поля:**

- `is_approved` (boolean) — одобрить или отклонить отзыв.

В спецификации указано `minProperties: 1` — нужно передать хотя бы одно поле (сейчас это и есть только `is_approved`).

---

### ReviewList

**Назначение:** формат ответа для GET /reviews — не «голый» массив, а объект с пагинацией.

**Поля:**

- `items` — массив объектов **Review** (отзывы на текущей странице).
- `total` — общее количество отзывов по применённым фильтрам; по нему считают количество страниц (например, total / limit).

---

### RatingSummary

**Назначение:** ответ GET /ratings/summary — готовые цифры для отображения рейтинга и числа отзывов.

**Поля:**

- `average` — средняя оценка (число; если отзывов нет, может быть `null`).
- `count` — количество отзывов, по которым посчитан рейтинг.
- `product_id` — для какого товара посчитано; `null` — рейтинг по магазину.

---

### Error

**Назначение:** единый формат ошибок по API_STANDARDS — все ошибки (400, 404, 422 и т.д.) возвращают объект с полями `error` и `code`, при необходимости — `details`.

**Поля:**

- `error` (обязательное) — текст ошибки.
- `code` (обязательное) — HTTP-код (400, 404, 422, 500 и т.д.).
- `details` (опционально) — объект с деталями (например, ошибки валидации по полям: `text`, `rating`).

---

## Переиспользуемые ответы (components/responses)

В спецификации три именованных ответа, чтобы не дублировать описание одних и тех же ошибок в каждом эндпоинте:

- **BadRequest** — «Некорректный запрос», код 400, тело по схеме **Error** (например, `"Некорректные данные"`).
- **NotFound** — «Ресурс не найден», код 404, тело по схеме **Error** (например, `"Отзыв не найден"`). Используется для GET/PATCH/DELETE по несуществующему `reviewId`.
- **ValidationError** — ошибка валидации данных, код 422, тело по схеме **Error**, при необходимости с полем `details` (ошибки по полям: пустой текст, неверный диапазон рейтинга и т.д.).

Они подключаются в эндпоинтах через `$ref`, чтобы описание было одно на весь файл.
